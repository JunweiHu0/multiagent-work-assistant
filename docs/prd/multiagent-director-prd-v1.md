- 日期：2026-07-09
- 状态：产品重新定案（头脑风暴收敛后的 V1 蓝图）
- 取代：`docs/prd/2026-07-09-command-console-prd.md`、`docs/prd/multiagent-work-assistant-prd.md`（桌宠视角）——两者的界面/桌宠框架被本文重置

---

## 1. 一句话

用户交一句抽象的任务,产品自动**组队 → 派活 → 盯场 → 验收返工**,把成品交回。用户只在**开头选阵容、结尾收货**时出现;中间的碎进展全部由一个"编排总监 agent"承担。

它不是聊天助手,不是编排框架,不是又一个任务看板。它是**你已经在用的那些闭源强 agent(CC / Codex / WorkBuddy…)之上的一个 AI 承包商/制片厂**——你是甲方,它是总包。

## 2. 目标用户

已经在用 CC、Codex、WorkBuddy 等桌面 agent 的人(开发者、技术创始人、重度 AI 协作者)。他们已经手动在多个 agent 之间切窗口、复制粘贴、盯进度——痛点是真实且自证的。产品的门槛不是"有没有用",是**"有没有惊喜"**:这群人一眼能想到"任务看板",不会被打动;能打动他们的是**"我说一句话,它甩给我三支配置好的团队让我挑,然后自己把活干完"**。

## 3. 核心价值与英雄镜头

**要解决的问题:** 一个人手动编排多个 AI agent,协调成本高到抵消了 AI 的杠杆——拆任务、分角色、写提示词、盯进度、判好坏、要返工,全是人肉。

**英雄镜头(V1 主打,用户已选定 = 选角):**

> 用户:「帮我做一个企业管理软件的库存模块」
> 总监:分析你手上的套餐与用量后,给出——
> **方案 A**:2× Claude Code 做开发 + 1× Codex 做产品/架构 + 1× WorkBuddy(GLM5.2)管数据库 · 预估耗时 ~3h · 预估配额:CC 40%、Codex 15% · 取舍:开发并行快,评审偏轻
> **方案 B**:1× Codex 开发 + 1× Claude Code 全程评审把关 + 1× GLM 数据库 · 慢但质量稳
> **方案 C**:…
> 用户挑 A(可微调)。之后总监新建各实例、写好角色提示词、派活、持续监督与评估、按需返工重派,**直到需求被验证实现**。全程用户可以去看电影。

这个"资源感知的选角提案"是全场最新鲜、最没人做过、最能当宣传片主画面的一块。**它同时是让"放手"变得可接受的心理扣子**:控制权交给用户的时机,恰好在钱还没花、事还没干的那一刻(选方案),选完用户才敢真的松手。

## 4. 产品原则

1. **放手型,不是监工型。** 默认全自动;用户介入是**例外**,不是每一步。
2. **例外必须弹回给用户**(逃生阀):总监真卡住 / 遇到不可逆且费钱的动作(花大钱、删东西、对外发布) / 超预算 / 自评置信度过低。
3. **开头选阵容是唯一的强制 checkpoint。** 这是控制权与信任的交接点。
4. **agent-neutral 贯彻到底**——包括总监自己的大脑:V1 总监走 API,用户自带 key、自选闭源模型(见 §6)。
5. **隐私是信任前提,不是合规项。** 只传元数据摘要,绝不读/存/传源码正文、diff、prompt、tool output、token、secret。power user 不会装一个会读他代码的东西。
6. **护栏即产品。** 预算闸、隔离沙箱、不可逆动作确认——不是对功能的限制,它们就是产品本身;第一次烧钱搞出一摊,用户就卸载。
7. **桌宠是 5% 的点缀,不是特色。** 核心永远是编排质量。桌宠是"你去看电影时,那个平时安静、只在总监需要你拍板时才亮起来"的 ambient 信号。

## 5. 核心概念（数据模型草案）

| 概念 | 说明 |
| --- | --- |
| **Brief(任务)** | 用户输入的抽象目标。 |
| **Director(编排总监)** | 大脑。做拆解、选角、派活、监督、评估、返工决策。V1 = LLM API。 |
| **Roster(可用资源)** | 用户已接入的 agent 及其套餐/配额/用量。选角的输入。 |
| **Team Option(团队方案)** | 一个候选阵容:角色 → agent 实例的映射 + 预估成本/配额/时长 + 取舍说明。产出 A/B/C。 |
| **Role(角色)** | 拆解出的职能(开发 / 架构-PM / 评审 / 数据库 / 测试…),含角色提示词。 |
| **Agent Instance(实例)** | 一个被拉起的执行体(某个 CC / Codex / WorkBuddy 会话),承担一个角色。 |
| **WorkPackage(工作包)** | 派给某实例的具体活,含依赖与验收口径。 |
| **Evaluation(评估)** | 总监对某产出的判定:通过 / 打回重做 / 重新分配。**产品天花板所在。** |
| **Escape(逃生阀事件)** | 需要用户介入的例外:stuck / irreversible / over-budget / low-confidence。 |
| **Delivery(交付)** | 成品 + 验收报告,交回用户做最终验收。 |

## 6. 系统架构（V1）

**两层分离——大脑与手,是本产品的架构骨架:**

```text
                    用户 Brief
                        │
        ┌───────────────▼───────────────┐
        │   Director（编排总监）= 大脑     │   V1: LLM API
        │   拆解 / 选角 / 派活 / 监督 /    │   用户自带 key, 自选闭源模型
        │   评估 / 返工决策 / 逃生判断      │   (BYO-API-brain)
        └───┬───────────┬───────────┬────┘
            │ 角色提示词 + 工作包        │
   ┌────────▼──┐   ┌────▼─────┐   ┌──▼──────────┐
   │ CC 实例×N  │   │ Codex 实例│   │ WorkBuddy/GLM│  = 手（执行体）
   │ (headless)│   │ (headless)│   │   实例        │  用户的套餐/配额
   └────┬──────┘   └────┬─────┘   └──┬──────────┘
        │  免token hooks → 事件(元数据)  │
        └───────────────▼───────────────┘
                 事件观测层（沿用协议）
                        │
        ┌───────────────▼───────────────┐
        │  主界面（编排监控室） + 桌宠(5%) │
        └────────────────────────────────┘
```

**大脑(Director):** V1 走 LLM API,**用户自带 key、自选各家闭源模型**。含义:①产品不自带模型,是"BYO-脑"的编排器,天然中立、成本归用户;②总监的判断力天花板部分由用户选的模型决定——想要更强的编排,用户可以把更强的模型摆在总监位。所以**围绕 API 的编排脚手架(拆解框架、评估 rubric、返工策略、逃生判据)的质量,是我们真正要打磨的核心资产**,而不是模型本身。

**手(执行体):** 用户已在用的桌面 agent,V1 优先走**无头入口**(`claude -p`、`codex exec` 等)以便可编程地"拉起实例 + 喂角色提示词 + 收产出";无头入口未知的(如 WorkBuddy/GLM)→ 先 probe,不确认前降级为"半自动:总监生成角色提示词,用户手动开实例粘贴"。

**观测层:** 沿用 agent-neutral 事件协议 —— 每个实例通过免 token hooks 把"它在干嘛"(元数据)回流,总监据此监督。这是重做也保留的金子。

## 7. 主界面信息架构（核心,非桌宠）

主界面 = **编排监控室**,三态:

1. **交办态**:一个输入框 + 用户的资源面板(有哪些 agent、配额剩多少)。用户敲抽象任务。
2. **选角态(英雄屏)**:总监返回的 A/B/C 团队方案,并排卡片,每张显示角色×实例映射、预估成本/配额/时长、取舍一句话。可微调(换某角色的 agent、增减实例)。一个[确认这支队]主按钮。**这一屏是产品的宣传片主画面,做到极致。**
3. **执行态**:实时编排盘 —— 顶部总监状态(在拆/在评/在等)、下面每个实例一张卡(角色、当前动作、产出、评估结果),实例间的依赖/交接可见;右侧"需要你处理"只在逃生阀触发时出现;底部事件流。用户想看时看,不想看就去干别的,桌宠替他盯。

## 8. 桌宠（5%,明确从属）

一个常驻、极小、不打扰的 ambient 存在。**唯一职责**:用户离开主界面(看电影、干别的)时,①用安静的状态灯表达"一切正常,在跑";②只在逃生阀触发(要你拍板/授权/超预算/卡住)时亮起来敲你一下。不承载任何编排逻辑,不做任何主操作。它是"你不看监控室时的那盏灯"。技术上是主应用的一个 always-on-top 小窗 + 托盘。

## 9. V1 范围（诚实收窄）

**V1 英雄 = 选角体验做到惊艳 + 一个"有边界的真实项目"被端到端自动跑完交付。**

必做:
- 资源感知的选角引擎:拆解 → A/B/C 团队方案(含配额/成本预估)→ 用户选/微调。
- 至少两类执行体的无头拉起 + 角色提示词派发 + 产出回收(Codex、CC 优先)。
- 监督-评估-返工闭环(在有边界的任务上可靠)。
- 四类逃生阀 + 护栏(预算闸、隔离 worktree/沙箱、不可逆动作确认)。
- 主界面三态 + 桌宠 5% ambient。
- 交付 + 验收报告。

V1 **不做 / 明确推迟**:
- "一句话做完整个企业级软件、隔夜全自动交付"——这是北极星/宣传愿景,V1 在**有边界的项目**上先跑通,不承诺无边界大项目的可靠自主。
- 产品自带模型(始终 BYO-API)。
- 云端 / 账号体系 / 团队协作 / 跨设备。
- 自动授权不可逆动作(永远要用户拍板)。
- 无头入口未 probe 通过的执行体的全自动(先半自动)。

## 10. 护栏 · 安全 · 成本

- **预算闸**:每个 Brief 设 token/花费上限,逼近即逃生阀弹回用户。
- **隔离执行**:执行体在 worktree / 沙箱里干活,不直接动用户真实主干,交付前用户可 diff。
- **不可逆动作门**:花大钱、删除、对外发布、改系统配置 → 永远弹回用户确认,不自动。
- **隐私**:观测只走元数据摘要;源码/prompt/diff/tool output/token/secret 绝不读存传。
- **BYO-API-key**:总监的 key 由用户提供、本地保存,产品不经手。

## 11. 关键风险与未验证假设（probe-first）

1. **总监判断力 = 产品天花板(P0)。** 拆解一个模糊大目标、评估产出好坏、决定收/返工/停止——这是当今自主多 agent 系统集体翻车处(漂移、绕圈、自信交垃圾)。缓解:①用户可选更强模型当总监;②结构化评估 rubric,而非放任 free-form;③"评审 agent"模式(用一个强 agent 当 QA 评判别人产出——这正是老代码里"Codex 写、Claude review"的自动化升级版);④逃生阀兜底。**V1 先在有边界任务上证明这个闭环靠谱,再谈放大。**
2. **执行体能否批量无头拉起(P0)。** `claude -p`、`codex exec` 可无头,但并发有各家限流/配额;WorkBuddy/GLM 无头入口未知。→ **先 probe,决定 V1 哪些全自动、哪些半自动。**
3. **成本/炸半径(P1)。** 全自动 + 自拉 agent + "干到完成"= 潜在无上限烧钱与乱改。→ 护栏(§10)是硬前提。
4. **拆解质量对任务规模敏感(P1)。** 越大越模糊越容易拆错。→ V1 收窄到有边界项目。

## 12. 成功标准（V1）

1. 用户敲一句有边界的真实任务,能拿到 ≥2 个"看得懂、敢选"的团队方案(含配额/成本预估)。
2. 选定后,总监能自动拉起实例、派活、并至少完成一轮"评估→返工"闭环,最终交付。
3. 全程用户中途介入次数 ≤ 逃生阀触发次数(即:不该被打扰的地方它没打扰)。
4. 护栏生效:一次都没有在用户不知情下花超预算 / 动真实主干 / 执行不可逆动作。
5. 隐私零泄漏:全链路无源码/prompt/token 外泄。
6. 主观:一个 CC/Codex 老用户看完选角屏 + 一次自动交付,说出"卧槽"。

## 附:我在这份 PRD 里替你做的几个假设（请你校准）

1. **执行体走无头 CLI(`claude -p`/`codex exec`)** 来实现"程序化拉起 N 个实例" —— 因为没法可编程地驱动 N 个 GUI 桌面窗口。如果你想要的是"驱动桌面 App 本身",那是另一套(且更难)的方案,得先说。
2. **总监(大脑)和执行体(手)是两层不同的东西**:大脑是你选的 API 模型(只思考、不写代码),手是 CC/Codex 实例(真干活)。我按这个理解写的。
3. **V1 收窄到"有边界的项目"**,把"一句话做完整个企业软件"当北极星而非 V1 承诺 —— 因为总监判断力是天花板,大而模糊的任务现在必翻车。如果你坚持 V1 就要挑战大项目,我们得单独聊怎么防它烧钱翻车。

这三条你认不认?认的话,下一步我可以把**选角引擎**(那个英雄屏背后的拆解→方案生成逻辑)单独展开成一份更细的功能规格,那是整个 V1 最该先想透的一块。
