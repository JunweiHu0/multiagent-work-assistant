# V1 构建顺序（依赖排期,非时间排期）

- 日期：2026-07-09
- 性质：**纯 vibe coding、持续执行的项目**——本文只排"先后依赖顺序",不排时间/日历/人天。执行者(Codex/CC)从本文取下一个未完成里程碑,构建到其"完成判据"(带测试),Fable review,遇到 GATE 停下让用户拍板,然后取下一个,循环不停。
- 产品源头：[../prd/multiagent-director-prd-v1.md](../prd/multiagent-director-prd-v1.md);里程碑映射见文末 roadmap 对照。

## 排序三原则

1. **先证伪,再铺量**:最先做最便宜、最能戳破风险假设的事(执行体无头探针)。
2. **英雄早出,且独立**:选角屏(hero)不依赖高风险的执行/评估闭环,可早做、早拿到第一个"卧槽"。
3. **护栏先于真实执行**:任何会花钱/改真实文件的派发之前,安全网必须已经立好(这细化了 roadmap 把"护栏"列在 V1.3 的排序——安全关键部分提前到派发之前)。

## GATE(即便持续执行也必须停)

- **产品 GATE**:S3 选角屏——英雄镜头,用户确认"就是这个感觉"再往下。
- **安全硬 GATE**:S4 未完成前,S5 真实派发绝不开工。
- **天花板 GATE**:S6 先在有边界任务反复证明靠谱,才谈放大;大而模糊任务 not-yet。
- **probe GATE**:S1 探针结论决定各执行体是"全自动"还是"半自动"。

### S0 — 地基 + 观测核心搬运

- **目标**:立起 V1 新代码骨架,并把 legacy 的观测层挖回来清干净。
- **建**:新 V1 源码根 + 运行时敲定(Node 做 glue/观测;director 大脑走 LLM API;console 栈按 PRD 倾向 React+shadcn+Electron,本里程碑定案);**BYO-API**(本地读用户 key + 选 vendor/model,不经手不上传);**Roster** 配置(用户声明手上有哪些 agent、席位/配额);**观测服务**——从 `legacy/orchestrator/relay.js` + `event-log.js` 挖回,de-entangle 掉 work-store,保留"逐字节透明 + JSONL + metadata-only + 先应答后转发"。
- **完成判据**:骨架能跑;能加载 key+model+roster;观测服务收事件并 metadata-only 落盘;冒烟测试绿。
- **不做**:不接 director、不派发。

### S1 — 执行体无头探针 · probe GATE

- **目标**:证实能不能无头拉起执行体、喂角色提示词、拿产出、并观测到。
- **建**:诊断脚本——`claude -p`、`codex exec` 各拉一实例,喂一段角色提示词跑一个安全小任务,捕获退出/产出,确认 hooks 事件回流观测服务;WorkBuddy/GLM 无头入口未知 → 探测。
- **完成判据**:产出 probe 结论文档,写明每个执行体:能否无头拉起、并发限流/配额观察、hooks 是否回流、哪些只能半自动。
- **GATE**:某执行体无头不可行 → 在选角里降级为"半自动"(生成提示词、用户手动开)。
- **红线**:只跑安全命令,不伪造事件,不读正文。

### S2 — 选角引擎(英雄大脑)

- **目标**:抽象任务 → 拆解角色 → roster/配额感知 → 生成 A/B/C 团队方案(角色×实例 + 配额/成本/时长预估 + 取舍一句话)。
- **建**:director(LLM API)的拆解脚手架 + 角色→执行体映射(依 S1 可行性)+ 配额/成本估算 + 方案生成器 → 结构化 A/B/C;评估 rubric 雏形。
- **完成判据**:给一个有边界真实任务,产出 ≥2 个"看得懂、字段完整、敢选"的方案。
- **不做**:不执行、不派发——纯规划产出。

### S3 — 选角屏(英雄 UI)· 产品 GATE

- **目标**:把 A/B/C 变成能点的界面:输入任务 → 看方案 → 挑/微调 → 确认一支队。
- **建**:薄 console(交办态 + 选角态两态);输出一个"已确认的编排计划"数据结构。
- **完成判据**:用户能敲任务、拿 A/B/C、挑、微调、确认。
- **GATE(产品)**:用户看完说"就是这个感觉"再往下——这是英雄镜头,值得停一下确认方向。

### S4 — 护栏基座 · 安全硬 GATE

- **目标**:任何真实执行前先立好安全网。
- **建**:隔离执行(worktree/沙箱,不动用户真实主干);预算闸(每任务 token/花费上限,逼近即停);不可逆动作门(花钱/删除/对外发布 → 弹回用户);逃生阀事件类型(stuck / over-budget / irreversible / low-confidence)。
- **完成判据**:一个模拟执行在沙箱里跑——超预算会停、不可逆动作被拦、逃生事件能产出。
- **红线**:护栏是硬前置,S5 之前必须过 S4。

### S5 — 无头派发

- **目标**:给定已确认的队,程序化拉起实例、写角色提示词、派发,产出回流观测。
- **建**:按 S1 结论用 `claude -p`/`codex exec` 拉起 N 实例,注入角色提示词与工作包,在 S4 的沙箱+预算+不可逆门保护下执行。
- **完成判据**:选一支队,真实拉起 ≥2 实例、各领角色提示词并开工、产出与事件回流观测;全程在护栏内。

### S6 — 监督-评估-返工闭环(产品天花板)· 天花板 GATE

- **目标**:总监观测产出、判定通过/打回/重派,直到需求验证。
- **建**:结构化评估 rubric(非放任)、"评审 agent"当 QA、返工/重派决策、**停止条件**(挡住无限循环烧钱)、置信度低 → 逃生。
- **完成判据**:有边界任务上至少完成一轮"评估→返工"并交付;无限循环被停止条件挡住;评估明显优于随机。
- **GATE(天花板)**:先在有边界任务反复证明靠谱,再谈放大。

### S7 — 执行态 console + 逃生处理

- **目标**:主界面第三态(执行监控室)+ 逃生阀弹回用户的交互。
- **建**:实时编排盘(总监状态 + 每实例卡 + 产出/评估 + 依赖交接);"需要你处理"区(授权/拍板/超预算)。
- **完成判据**:用户全程只看这一屏掌握进度,逃生触发时在此拍板/授权。

### S8 — 桌宠 5% ambient

- **目标**:常驻置顶态势灯,平时安静,只在逃生阀触发时亮起来敲你一下。
- **建**:always-on-top 小窗 + 托盘,订阅逃生事件,不承载任何编排逻辑。
- **完成判据**:用户离开主界面时一切正常则安静,有逃生事件则提醒。

### S9 — V1 端到端验收 · V1 成

- **目标**:一个有边界真实任务,交办→选角→挑→派发→监督评估返工→交付,全程跑通。
- **完成判据**:满足 PRD §12 六条(≥2 方案敢选、一轮评估返工后交付、介入次数≤逃生次数、护栏零失守、隐私零泄漏、老用户"卧槽")。

## Beyond V1（不排入 V1 顺序,后续）

- MCP 化让大脑可插拔(用户任意 agent 当总监)。
- 第三个执行体 adapter(WorkBuddy/GLM 或 Cursor)证明联合性。
- 放大到更大 / 更模糊的项目。

## 持续执行怎么转

执行者取下一个未完成 Sx → 构建到"完成判据"(带测试)→ Fable review → 命中 GATE 则停下让用户拍板,否则继续取下一个。**每个 Sx 独立可验证**,不靠时间靠判据。红线贯穿全程:probe-first、隐私 metadata-only、不自动授权不可逆动作、BYO-API。
